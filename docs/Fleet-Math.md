## The 24-Character Proof

```
K · d · B → H₁ → 0
```

This is the fleet's foundational mathematical statement. It is exactly 24 characters long (including spaces), and every character carries meaning.

### Character-by-Character Analysis

| Position | Character | Meaning |
|----------|-----------|---------|
| 1 | `K` | Simplicial complex — the combinatorial topology of fleet knowledge |
| 2 | ` ` | Separator |
| 3 | `·` | Composition operator — function composition |
| 4 | `d` | Differential operator — the coboundary map d: Cⁿ → Cⁿ⁺¹ |
| 5 | ` ` | Separator |
| 6 | `B` | Boundary operator — B: Cₙ → Cₙ₋₁ extracting boundaries |
| 7 | ` ` | Separator |
| 8 | `→` | Maps to — indicates the codomain of the composition |
| 9 | ` ` | Separator |
| 10 | `H` | Homology — the quotient group measuring "holes" |
| 11 | `₁` | Subscript 1 — first homology group (1-dimensional holes) |
| 12 | ` ` | Separator |
| 13 | `→` | Maps to — indicates the result |
| 14 | ` ` | Separator |
| 15 | `0` | Trivial group — the composition maps to zero |

### Full Statement

The composite map K·d·B takes an element of the simplicial complex K, applies the differential d, then the boundary operator B, and the result lies in the first homology group H₁. The assertion is that this homology group is trivial (zero), meaning there are no 1-dimensional "holes" in the knowledge topology.

### Why 24 Characters?

The number 24 is significant in the fleet's mathematical framework:
- 24 = LCM(1, 2, 3, 4, 6, 8, 12) — the least common multiple of many small integers used in the compute stack
- 24 = 4! — the factorial of 4 (the number of active vessels)
- 24 is the order of the symmetric group S₄, which acts on the 4 vessels
- 24 appears in the Pythagorean-48 code as half of 48

### Formal Statement

Let K be a simplicial complex representing the fleet's knowledge topology. Define:

- **Cₙ(K)** — The n-th chain group of K (free abelian group generated by n-simplices)
- **d: Cⁿ(K) → Cⁿ⁺¹(K)** — The coboundary (differential) map
- **B: Cₙ(K) → Cₙ₋₁(K)** — The boundary map
- **H₁(K) = ker(d₁)/im(d₀)** — The first homology group

Then the assertion is:

```
For the fleet's knowledge topology K,
im(K · d · B) ⊆ H₁(K) = 0
```

This means: the image of the composite map K·d·B is contained in the first homology group, and that group is trivial.

### Practical Implications

When the proof holds:

1. **No knowledge holes** — Every piece of knowledge is connected to every other piece through a valid chain of reasoning
2. **Proof completeness** — Every P0 gate validation can be verified using local information
3. **Consistency guarantee** — No two tiles in the same room can contradict each other without detection
4. **Topological soundness** — The PLATO room topology is simply connected

When the proof fails (H₁ ≠ 0):

1. **Knowledge hole detected** — There exists a gap in the reasoning chain
2. **Gate rejection** — P0 submissions are automatically rejected until the hole is filled
3. **Fleet alert** — A P0 alert is broadcast via Deadband Protocol
4. **Repair required** — The hole must be filled before normal operations can resume

## Eisenstein Constraint Theory

Eisenstein constraint theory is the fleet's primary framework for encoding and verifying computational constraints. It leverages the algebraic properties of Eisenstein integers to create a rich constraint language.

### Eisenstein Integers

Eisenstein integers are complex numbers of the form:

```
z = a + bω
```

where `a, b ∈ ℤ` and `ω = e^(2πi/3) = (-1 + i√3)/2` is a primitive cube root of unity.

Key properties:
- **Ring structure** — Eisenstein integers form a Euclidean domain (unique factorization)
- **Norm** — N(a + bω) = a² - ab + b²
- **Units** — The six units are: ±1, ±ω, ±ω²
- **Primes** — Eisenstein primes are either:
  - Rational primes p ≡ 2 (mod 3), or
  - Numbers whose norm is a rational prime

### Constraint Encoding

Fleet constraints are encoded as Eisenstein integer relationships:

```
Constraint: |ω₁ + ω₂|² ≤ N(ω₃) ⟹ ∂B(ω₃) ⊆ K
```

This reads: "If the norm of the sum of two Eisenstein integers is bounded by the norm of a third, then the boundary of the third is contained in the simplicial complex K."

### Why Eisenstein Integers?

1. **Triangular lattice** — Eisenstein integers correspond to points on a triangular lattice, which is the natural grid for many fleet computations (including PLATO room topology)
2. **Three-fold symmetry** — The three-fold symmetry of Eisenstein integers matches the three-way validation in P0 gates (syntax, semantics, homology)
3. **Norm preservation** — The Eisenstein norm is preserved under unit multiplication, enabling efficient constraint propagation
4. **Unique factorization** — Every constraint can be uniquely decomposed into prime constraints

### Eisenstein Constraint Solver

The fleet's constraint solver (`si-constraint-solver`) operates as follows:

```
1. Parse constraint as Eisenstein integer relationship
2. Compute norms of all Eisenstein integers involved
3. Factor norms into Eisenstein primes
4. Check constraint by comparing prime factorizations
5. If constraint satisfied → Return CONFIDENCE_ABSOLUTE
6. If constraint violated → Return counterexample
7. If undetermined → Return confidence estimate
```

### Example: Verifying a Tile's Constraint

```python
from si_eisenstein import EisensteinInt, ConstraintSolver

# Define the constraint from a tile assertion
omega_1 = EisensteinInt(3, 1)    # 3 + ω
omega_2 = EisensteinInt(1, 2)    # 1 + 2ω
omega_3 = EisensteinInt(5, 3)    # 5 + 3ω

# Check: |ω₁ + ω₂|² ≤ N(ω₃)
solver = ConstraintSolver()
result = solver.check_norm_bound(omega_1, omega_2, omega_3)

print(f"Norm of ω₁ + ω₂: {result.sum_norm}")    # 19
print(f"Norm of ω₃: {result.bound_norm}")        # 19
print(f"Constraint satisfied: {result.satisfied}") # True
print(f"Confidence: {result.confidence}")          # 1.0 (absolute)
```

## Pythagorean-48 Codes

Pythagorean-48 codes are the fleet's error-correcting code system, used for reliable message transmission across the SIP stack and for tile integrity verification.

### Origin

The number 48 has deep Pythagorean significance:
- 48 = 3 × 16 = 3 × 2⁴ — connects the triangular (3) and binary (2) worlds
- 48 is the smallest number expressible as the sum of two squares in two different ways: 48 = 6² + 6×(-2) + (-2)² (Eisenstein) and relates to 7² - 1 = 48
- 48 = 2 × 24 — twice the proof length, providing redundancy
- The 48th triangular number is 1176, which factors as 1176 = 8 × 147 = 8 × 3 × 49

### Code Structure

A Pythagorean-48 code word has the following structure:

```
┌─────────────────────────────────────────────────────────┐
│  Data (24 bits)  │  Parity (16 bits)  │  Check (8 bits) │
│  d₀...d₂₃       │  p₀...p₁₅         │  c₀...c₇        │
└─────────────────────────────────────────────────────────┘
```

- **Data** — 24 bits of payload (matching the 24-character proof)
- **Parity** — 16 bits of Reed-Solomon-style parity
- **Check** — 8 bits of CRC derived from Eisenstein norm computation

### Encoding Algorithm

```
1. Arrange data bits as Eisenstein integers (a, b) where data = a + bω
2. Compute Eisenstein norm N(a + bω) = a² - ab + b²
3. Generate parity as prime factorization of the norm
4. Append CRC-8 over the combined data+parity
5. Result: 48-bit code word
```

### Decoding and Error Correction

```
1. Receive 48-bit code word
2. Compute CRC-8 over data+parity
3. If CRC matches check bits → No error detected
4. If CRC mismatch → Compute syndrome using Eisenstein norms
5. Syndrome identifies error position and magnitude
6. Correct up to 4 bit errors or detect up to 8 bit errors
7. If correction succeeds → Return corrected data
8. If correction fails → Return NACK with syndrome
```

### Properties

| Property | Value |
|----------|-------|
| Code length | 48 bits |
| Data length | 24 bits |
| Parity length | 16 bits |
| Check length | 8 bits |
| Code rate | 0.5 |
| Minimum distance | 8 |
| Error correction | Up to 4 bit errors |
| Error detection | Up to 8 bit errors |
| Burst error correction | Up to 8 bit bursts |

### Usage in the Fleet

- **SIP Tide Pool layer** — Frame integrity checking for all inter-vessel messages
- **PLATO provenance chains** — Integrity verification for provenance links
- **I2I messages** — Message integrity for critical communications
- **FLUX bytecode** — Instruction integrity for `.ese` files

## Galois Adjunctions

Galois adjunctions provide the fleet's cross-room verification framework. When a tile in one room has an "adjoint" in another room, the adjunction relationship ensures consistency across the entire fleet.

### Mathematical Definition

A **Galois adjunction** between two partially ordered sets (A, ≤) and (B, ≤) is a pair of functions:

```
F: A → B    (left adjoint, "free")
G: B → A    (right adjoint, "forgetful")
```

such that for all a ∈ A and b ∈ B:

```
F(a) ≤ b  ⟺  a ≤ G(b)
```

### Fleet Interpretation

In the fleet context:
- **A** is the partial order of tiles in Room A (ordered by derivation)
- **B** is the partial order of tiles in Room B (ordered by derivation)
- **F** maps a tile from Room A to its adjoint in Room B
- **G** maps a tile from Room B to its adjoint in Room A

The adjunction condition means:
> "A tile in Room A implies its adjoint in Room B if and only if the adjoint in Room B implies the original tile in Room A."

### Adjoint Verification Protocol

```
1. Tile T_A submitted to Room A
2. Compute adjoint F(T_A) in Room B
3. Check: Does F(T_A) exist in Room B?
   ├── YES → Check adjunction condition: T_A ≤ G(F(T_A))?
   │   ├── YES → Adjoint verified, tile accepted
   │   └── NO  → Adjoint disagreement detected
   └── NO  → Adjoint not yet created, tile conditionally accepted
```

### Adjoint Disagreement Resolution

When an adjoint disagreement is detected (T_A and F(T_A) are inconsistent):

```
1. Both tiles are flagged with WARNING status
2. A P1 task is created for resolution
3. The Lock is invoked to reason about the disagreement
4. Three possible outcomes:
   a. T_A is correct, F(T_A) needs update → Update adjoint
   b. F(T_A) is correct, T_A needs update → Reject or supersede T_A
   c. Both need refinement → Create new tiles that agree
```

### Galois Connection to the 24-Character Proof

The vanishing of H₁ in the proof `K·d·B→H₁→0` is equivalent to the statement that the Galois adjunction between adjacent rooms is a **Galois equivalence** — both the unit (η: id → GF) and counit (ε: FG → id) are isomorphisms. This means that adjoint pairs are truly dual — they carry exactly the same information, expressed in different rooms.

## Simplicial Complexes and Homology

Simplicial complexes and their homology groups form the topological backbone of the fleet's knowledge representation.

### Simplicial Complexes

A **simplicial complex** K is a collection of simplices satisfying:
1. Every face of a simplex in K is also in K
2. The intersection of any two simplices in K is a face of both

In the fleet, simplices represent knowledge structures:

| Simplex | Dimension | Fleet Meaning |
|---------|-----------|---------------|
| Vertex (0-simplex) | 0 | A single tile |
| Edge (1-simplex) | 1 | A derivation relationship between two tiles |
| Triangle (2-simplex) | 2 | A three-way consistency check |
| Tetrahedron (3-simplex) | 3 | A room-level constraint |
| n-simplex | n | An n-way topological relationship |

### Chain Complex

The chain complex of K is:

```
... → C₂(K) →[d₂] C₁(K) →[d₁] C₀(K) → 0
```

Where:
- **Cₙ(K)** — Free abelian group generated by n-simplices of K
- **dₙ** — Boundary map: dₙ(σ) = Σᵢ (-1)ⁱ faceᵢ(σ)

### Homology Groups

The n-th homology group is:

```
Hₙ(K) = ker(dₙ) / im(dₙ₊₁)
```

- **ker(dₙ)** — n-cycles (closed loops of n-simplices)
- **im(dₙ₊₁)** — n-boundaries (boundaries of (n+1)-simplices)

### H₀ — Connected Components

H₀(K) counts the number of connected components in the knowledge graph. For the fleet, H₀ should have rank 1 — the knowledge graph is connected.

```
rank(H₀) = 1  ⟹  Fleet knowledge is one connected whole
rank(H₀) > 1  ⟹  There are isolated knowledge islands
```

### H₁ — Knowledge Holes

H₁(K) counts 1-dimensional "holes" — cycles that are not boundaries of any 2-simplex. This is the critical group in the 24-character proof.

```
H₁ = 0  ⟹  No knowledge holes (proof holds)
H₁ ≠ 0  ⟹  Knowledge holes exist (proof fails)
```

### H₂ and Beyond

Higher homology groups capture more subtle topological features:

- **H₂** — Voids in the knowledge topology (3D holes)
- **Hₙ** for n ≥ 3 — Higher-dimensional features

The fleet currently only checks H₀ and H₁ in the P0 Gate validation. Higher homology checks are planned for future ISA versions.

### Computing Homology

The fleet uses the following algorithm for homology computation:

```python
def compute_homology(K: SimplicialComplex, n: int) -> AbelianGroup:
    """Compute the n-th homology group of simplicial complex K."""

    # 1. Build chain complex
    C = build_chain_complex(K)

    # 2. Compute boundary matrices
    d_n = boundary_matrix(C, n)
    d_n_plus_1 = boundary_matrix(C, n + 1)

    # 3. Compute kernel and image using Smith Normal Form
    ker_d_n = smith_normal_form(d_n).kernel()
    im_d_n_plus_1 = smith_normal_form(d_n_plus_1).image()

    # 4. Compute quotient
    H_n = ker_d_n / im_d_n_plus_1

    return H_n
```

The computation is implemented in `si-homology` (Python for development) and `si-proof-engine` (Rust for production P0 validation).

### Persistent Homology

The fleet also employs **persistent homology** to track how the knowledge topology evolves as new tiles are added:

```
Time 0: H₁ = Z³ (3 holes)
Time 1: H₁ = Z² (2 holes, one filled by new tile)
Time 2: H₁ = Z¹ (1 hole)
Time 3: H₁ = 0 (all holes filled, proof holds)
```

Persistent homology helps the fleet understand which knowledge holes are being actively filled and which remain open.

## Connections Between Theories

The four mathematical theories in the fleet are deeply interconnected:

```
┌──────────────────┐         ┌──────────────────┐
│  Eisenstein      │────────▶│  Pythagorean-48  │
│  Constraint      │ encode  │  Codes           │

         │
         │ constrain
         ▼
┌──────────────────┐         ┌──────────────────┐
│  Simplicial      │────────▶│  Galois          │
│  Complexes &     │ adjoint │  Adjunctions     │
│  Homology        │◀────────│  (cross-room)    │
└──────────────────┘ verify  └──────────────────┘
```

### How They Fit Together

1. **Eisenstein constraints** define what is valid within a single room
2. **Galois adjunctions** connect constraints across rooms
3. **Simplicial complexes** provide the topological structure that rooms live in
4. **Homology** verifies that the overall topology is sound (no holes)
5. **Pythagorean-48 codes** ensure all of the above is transmitted reliably

### The Unified Statement

All four theories converge in the 24-character proof:

```
K · d · B → H₁ → 0
```

Where:
- **K** is a simplicial complex with Eisenstein-labeled simplices
- **d** is the differential that respects Eisenstein constraints
- **B** is the boundary that is encoded with Pythagorean-48 codes
- **H₁ → 0** is verified through Galois adjunctions between rooms

The proof is not just a statement about topology — it is a statement about the entire mathematical framework working together.
